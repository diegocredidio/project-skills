# cc-flow Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Ship `cc-flow` + `cc-sync` skills (2 new skills), 10 starter template files, `docs/guardrails.md`, and wire cc-flow into existing skills (pm-handoff, kanban-pickup) + README + flow map + plugin manifest + changelog.

**Design doc:** `docs/plans/2026-04-19-cc-flow-design.md` — read before starting.

**Architecture:** Read-only over planning artifacts; writes only to `.claude/agents/<feature>/` (consumer) + `.cc-templates/` (first-run) + `.claude/.cc-manifest.md` + `.pm/cc-config.md`. Mirror of kanban-sync's pattern (pre-flight → preview → phased writes → incremental manifest).

---

## Conventions

Same as prior plans in this repo:

1. Read relevant files (existing skills for context, this plan, design doc).
2. Apply edits (Edit for surgical, Write for new files).
3. Verify with `grep` commands specified per task.
4. Commit with imperative message specified per task.

**Reference skill patterns:**

- For `cc-flow` — mirror `skills/kanban-flow/SKILL.md` (setup vs cycle modes, config file, mode dispatch).
- For `cc-sync` — mirror `skills/kanban-sync/SKILL.md` (pre-flight checks, dry-run preview with [CREATE]/[UPDATE]/[SKIP] markers, phased execution, incremental manifest saves, archive-don't-delete).

**Template syntax convention** (applies to all template files):

- Interpolation: `{{var}}` — cc-sync substitutes from artifacts.
- Partials: `{{> partial-name}}` — cc-sync includes from `templates/partials/<name>.md.tmpl`.
- No `{{#if}}`/`{{#each}}` — branching done by cc-sync choosing which partials to include, not in-template logic.
- Available vars documented in cc-sync SKILL.md and `templates/README.md`.

## Task 2: Create `skills/cc-sync/SKILL.md`

**New file.** This is the biggest skill file in the pack — read kanban-sync carefully first.

**Step 1:** Read `skills/kanban-sync/SKILL.md` end-to-end.

**Step 2:** Create `/Users/diegocredidio/Documents/repogit/project-skills/skills/cc-sync/SKILL.md`.

**Frontmatter (verbatim):**
```
---
name: cc-sync
description: Renders planning artifacts into per-feature Claude Code agents at `.claude/agents/<feature>/`, using templates from `.cc-templates/` as source. Reads PROJECT_PROFILE.md, PRD, ARCHITECTURE, STACK, TOKENS, COMPONENT_SPECS, API, ROUTES, QA_STRATEGY, TEST_CASES (whichever exist per feature). Idempotent — maintains `.claude/.cc-manifest.md` with SHA-256 of every owned file, uses checksum-prompt policy when user-edited files diverge, archives (not deletes) agents whose source feature was removed upstream. Use when cc-flow invokes it or when user says "sync the guardrails", "update agents for <feature>". Not for editing artifacts, not for executing tests, not for writing implementation code.
---
```

**Body structure** (mirror kanban-sync):

- H1 + 1-line purpose
- `## Pre-flight checks (ABORT if any step fails)`
  - Confirm `.pm/cc-config.md` exists; else abort with "run cc-flow setup first"
  - Confirm `.cc-templates/agents/` exists; else abort
  - Confirm a feature slug has been resolved (either passed by cc-flow or asked from user)
  - Confirm `.pm/<feature>/PROJECT_PROFILE.md` exists; else abort with "run pm-handoff or pm-grill first"
- `## How to run`
  - Step 1: Load inputs
    - `.pm/<feature>/PROJECT_PROFILE.md` (MUST), PRD (if exists), ARCHITECTURE (if exists)
    - `.design/<feature>/TOKENS.md`, `COMPONENT_SPECS.md`, `IA.md` (if exist)
    - `.backend/<feature>/BACKEND_STACK.md`, `BACKEND_DATA.md`, `BACKEND_API.md` (if exist)
    - `.frontend/<feature>/FRONTEND_STACK.md`, `FRONTEND_ROUTES.md`, `COMPONENT_PLAN.md` (if exist)
    - `.qa/<feature>/QA_STRATEGY.md`, `TEST_CASES.md` (if exist)
    - `.claude/.cc-manifest.md` (if exists — for diff)
  - Step 2: Extract template variables
    - List every variable cc-sync extracts — document the mapping artifact → var
    - Example: `{{stack.framework}}` ← row "Framework" in FRONTEND_STACK.md decision table
    - Include `{{artifactPaths.tokens}}`, `{{artifactPaths.componentSpecs}}` etc. (absolute paths for agents to reference)
  - Step 3: Select partials
    - designMode=shadcn-theme → include `shadcn-theme-notes` partial in frontend-engineer render
    - designMode=custom-system → include `custom-system-notes`
    - testingRigor=mvp → include `testing-rigor-mvp` in qa-engineer render
    - testingRigor=full → include `testing-rigor-full`
  - Step 4: Render templates
    - For each agent template (backend / frontend / qa / product-architect / design-reviewer) in `.cc-templates/agents/`:
      - Substitute `{{var}}` with extracted values
      - Resolve `{{> partial-name}}` by reading `.cc-templates/partials/<name>.md.tmpl` and substituting vars
      - Compute SHA-256 of rendered content
  - Step 5: Dry-run preview
    - For each rendered file, classify against manifest:
      - `[CREATE]` — file not in manifest, doesn't exist on disk
      - `[UPDATE]` — file in manifest, SHA matches manifest → will overwrite
      - `[PROMPT]` — file in manifest, SHA differs from manifest → user-edited, prompt
      - `[SKIP-DELETED]` — file in manifest but missing on disk → log, don't regenerate
    - Show user the preview table
    - Ask: "Proceed? (y/n)"
  - Step 6: Execute
    - For each file based on classification:
      - `[CREATE]` / `[UPDATE]`: write, compute SHA, append to manifest
      - `[PROMPT]`: ask user "keep edits / overwrite", act accordingly
      - `[SKIP-DELETED]`: skip silently, keep in manifest
    - Save manifest incrementally (after each write)
  - Step 7: Archive pass
    - For each file in manifest whose feature is no longer in `.pm/`/`.design/`/`.backend/`/`.frontend/`/`.qa/`:
      - Move file to `.claude/archive/<timestamp>/<feature>/<path>`
      - Update manifest row to Archived section
  - Step 8: Report
    - Summary: N created, M updated, K prompted, X archived, Y skipped
    - Pointer: "Invoke `@<feature>-backend-engineer` etc. in Claude Code to use the generated agents."
- `## Template variable reference`
  - Full table of every `{{var}}` cc-sync supports, with source artifact and field reference
- `## The .claude/.cc-manifest.md file`
  - Schema with 3 sections (Owned, Archived, User-modified)
- `## Rules`
  - Manifest is authoritative for ownership — never touch files not in it
  - Manifest saves incrementally, not batched, to survive partial failures
  - Archive, don't delete — move to `.claude/archive/<timestamp>/`
  - User-edited files: checksum-prompt always; never overwrite without explicit user confirmation
  - Read-only over artifact trees; writes only to `.claude/` and its sub-directories
  - First sync: smoke test with 1 agent first; abort cleanly if Claude Code frontmatter validation fails

**Step 3:** Verify:
- File exists.
- `grep -n "cc-manifest" skills/cc-sync/SKILL.md` → at least 4 matches.
