## Context

Adding a BDD workflow to the pack. Two new skills (`bdd-flow` + `bdd-scenarios`) and four modified existing skills (`pm-handoff`, `backend-brief-intake`, `frontend-brief-intake`, `qa-cases`). BDD is optional — teams that don't use it see no change.

Key design decisions (read the design doc for full rationale):
- `bdd-flow` is a thin orchestrator; `bdd-scenarios` does all the heavy lifting
- Output lands in `.bdd/<feature>/BACKEND_FEATURES.md` + `FRONTEND_FEATURES.md`
- `bdd-scenarios` conducts the Three Amigos session interactively, per FR
- Lineage-aware: reads parent BDD files when `PARENT.md` exists
- `qa-cases` detects BDD files and switches to augment mode (no scenario rewriting)

The skill must:

**Frontmatter:**
```yaml

name: bdd-flow
description: Conducts a Three Amigos BDD session for a named feature, producing `.bdd/<feature>/BACKEND_FEATURES.md` and `.bdd/<feature>/FRONTEND_FEATURES.md` — Gherkin scenarios separated by discipline (API-level vs UI-level). **Title:** `# BDD Flow — Three Amigos Scenario Session`
```

**One-line intro:** "Orchestrates the BDD scenario session for a feature, then hands back."

**How to run:**

Step 1: Verify inputs
- Ask user: "Which feature? (name of the `.pm/<feature>/` folder)"
- Check that `.pm/<feature>/PRD.md` exists. If missing: abort, tell user to run `pm-prd` first.
- Check that `.pm/<feature>/ARCHITECTURE.md` exists. If missing: abort, tell user to run `pm-architecture` first.
- Check for `.pm/<feature>/PARENT.md` — note presence for lineage mode (pass to bdd-scenarios).

Step 2: Create output folder
- Create `.bdd/<feature>/` if it does not exist.

Step 3: Invoke `bdd-scenarios`
- Tell user: "Starting Three Amigos session. I'll work through each FR with you."
- Invoke `bdd-scenarios` with the feature name and lineage flag.

Step 4: Hand back
- After `bdd-scenarios` completes, tell user: "Feature files written to `.bdd/<feature>/`. You can now run `backend-flow` and `frontend-flow` — both will read these as behavioral contracts."

**Rules:**
- Never write scenarios directly — that belongs to `bdd-scenarios`.
- Never proceed without PRD.md and ARCHITECTURE.md.
- Always create `.bdd/<feature>/` before invoking `bdd-scenarios`.

## Phase 2 — `bdd-scenarios` Three Amigos conductor

**Task #2 — Create `skills/bdd-scenarios/SKILL.md`**

This is the central skill. Create `skills/bdd-scenarios/SKILL.md`.

**Frontmatter:**
```yaml
---
name: bdd-scenarios
description: Conducts an interactive Three Amigos BDD session for a named feature. Reads PRD.md, ARCHITECTURE.md, and (if lineage) parent BDD feature files, then facilitates a per-FR question-and-answer session covering PM, Dev, and QA perspectives. Produces `.bdd/<feature>/BACKEND_FEATURES.md` (API-level Gherkin) and `.bdd/<feature>/FRONTEND_FEATURES.md` (UI-level Gherkin). Invoked by bdd-flow. Not for standalone use — use bdd-flow as the entry point.
---
```

**Title:** `# BDD Scenarios — Three Amigos Session Conductor`

**How to run:**

### Step 1: Load inputs

Read:
- `.pm/<feature>/PRD.md` — FR-XXX list with priorities and user journeys
- `.pm/<feature>/ARCHITECTURE.md` — system boundaries, API shape, layers

If `.pm/<feature>/PARENT.md` exists (lineage mode):
- Read the H1 to extract `<parent-slug>`
- Check for `.bdd/<parent-slug>/BACKEND_FEATURES.md` — load if present
- Check for `.bdd/<parent-slug>/FRONTEND_FEATURES.md` — load if present
- If parent BDD files exist: announce "Lineage detected. Parent `<parent-slug>` has existing BDD scenarios — I'll only run the session for new or modified FRs."
- If parent BDD files are absent: announce "Lineage detected but parent `<parent-slug>` has no BDD artifacts — running full session for all FRs."

### Step 2: Scope the session

Build the FR list from PRD.md.

If lineage and parent BDD files exist:
- Identify FRs from the PRD `## 0. Extends` table marked as `unchanged` → exclude from session (inherited)
- Identify FRs marked as `extends`, `modifies`, or `new` → include in session
- Show the user a summary:
  ```
  Session scope:
  ✅ Inherited (no session needed): FR-001, FR-002
  🔄 To cover: FR-003 (modifies), FR-004 (new), FR-005 (new)
  ```

If no lineage (or no parent BDD files): include all FRs.

Group FRs by user journey when possible — batch FRs that share a journey into one session round.

### Step 3: Announce session structure

Tell the user:
```
I'll work through [N] FRs (or FR groups). For each one I'll ask three rounds of questions:
  1. PM perspective — business rules and boundaries
  2. Dev perspective — contracts and constraints
  3. QA perspective — negative paths and edge cases

Then I'll draft backend and frontend scenarios for your review. Let's start.
```

### Step 4: Three Amigos session (repeat per FR or FR group)

For each FR (or group):

**4a. Present the FR:**
```
--- FR-XXX: [Title] ---
[Description from PRD]
Priority: [must / should / could]
```

**4b. PM perspective questions:**
Ask exactly:
> "PM lens — What is the exact business rule here? What must NOT happen (the boundary that, if crossed, means the feature is broken)?"

Wait for answer. If the answer is vague (e.g., "it should work correctly"), ask one follow-up:
> "Can you be more specific? For example: 'User can only do X if Y is true' or 'The system must reject Z under condition W'."

**4c. Dev perspective questions:**
Ask exactly:
> "Dev lens — What is the expected API contract (endpoint, method, key request/response fields)? Any technical constraint or invariant the implementation must respect?"

Wait for answer. One follow-up if vague.

**4d. QA perspective questions:**
Ask exactly:
> "QA lens — What is the critical negative path? What's the edge case that always gets forgotten for this kind of feature?"

Wait for answer. One follow-up if vague.

**4e. Draft scenarios:**
From the three answers, draft:
- 1–3 backend scenarios (API-level: request → response)
- 1–3 frontend scenarios (UI-level: user action → visible outcome)

Use canonical Gherkin shape:
```gherkin
Scenario: [Short descriptive name — one behavior]
  Given [starting context]
  When [the action under test]
  Then [observable expected outcome]
```

Backend scenarios: request/response oriented, no UI references, use HTTP verbs and status codes.
Frontend scenarios: user-action oriented, no API endpoint references, outcomes are visible UI changes.

**4f. Present drafts and iterate:**
Show:
```
Backend scenarios (API-level):
[scenarios]

Frontend scenarios (UI-level):
[scenarios]

Look good? (yes / suggest changes)
```

If user suggests changes: revise and re-present. Repeat until user says yes.

**4g. Confirm and move on:**
Once approved, note the scenarios as final and move to the next FR.

### Step 5: Handle inherited scenarios (lineage only)

If lineage mode with parent BDD files:
For FRs classified as `unchanged`:
- Do NOT generate new scenarios
- Record a reference line for the output file:
  `<!-- FR-XXX: inherited from <parent-slug> — see .bdd/<parent-slug>/BACKEND_FEATURES.md -->`

### Step 6: Write BACKEND_FEATURES.md

Save to `.bdd/<feature>/BACKEND_FEATURES.md`:

````markdown
# Backend Feature Scenarios: [Feature Name]

**Feature:** [feature slug]
**Source:** PRD.md, ARCHITECTURE.md
**Date:** [YYYY-MM-DD]
**Lineage:** [standalone | evolution of <parent-slug>]

---

## FR-XXX: [FR Title]

```gherkin
Feature: [Feature name]

  Scenario: [name]
    Given [context]
    When [action]
    Then [outcome]
```

<!-- FR-YYY: inherited from <parent-slug> — see .bdd/<parent-slug>/BACKEND_FEATURES.md -->

---

## Coverage index

| FR | Scenarios |
|----|-----------|
| FR-XXX | [scenario names] |
| FR-YYY | inherited |
````

### Step 7: Write FRONTEND_FEATURES.md

Save to `.bdd/<feature>/FRONTEND_FEATURES.md` with the same structure, using UI-level scenarios.

### Step 8: Hand back to bdd-flow

Report: "Session complete. [N] FRs covered, [M] inherited from parent. Files written."

**Rules:**
- Never skip the three-perspective question sequence for any FR in scope.
- Never guess on vague answers — always ask one follow-up.
- Backend scenarios: never reference UI elements. Frontend scenarios: never reference API endpoints.
- Inherited FRs get reference comments only — never copy-paste parent scenarios into child files.
- Coverage index at the end of each file is mandatory.
- Session proceeds FR by FR, waiting for user approval at each step before moving on.

```
### Step 2.6: BDD contract check

Check for `.bdd/<feature>/BACKEND_FEATURES.md`.
```

```
### Step 1.5: Detect mode

If both `.bdd/<feature>/BACKEND_FEATURES.md` AND `.bdd/<feature>/FRONTEND_FEATURES.md` exist:
→ **Augment mode**: scenarios already written by bdd-scenarios. Skip Step 4 (Write Gherkin scenarios).
  Instead:
  - Import FR-ID anchors from both files to build the coverage matrix (Step 5)
  - Add edge cases and negative cases not already covered in the BDD files (append to TEST_CASES.md under "Additional edge cases")
  - Write flakiness notes as usual (Step 6)
  - In TEST_CASES.md header, note: "Scenarios sourced from `.bdd/<feature>/` (Three Amigos session)"

If BDD files are absent:
→ **Standard mode**: continue with Steps 2–8 as written (current behavior unchanged).
```

**Update TEST_CASES.md template header** to include an optional `**Scenario source:**` line:
```
**Scenario source:** [bdd-flow Three Amigos session | derived from PRD + QA_STRATEGY]
```
